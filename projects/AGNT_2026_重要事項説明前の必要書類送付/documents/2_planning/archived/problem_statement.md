# 課題定義書

**プロジェクト名**: AGNT 2026 重要事項説明前の必要書類送付

## 1. 概要

本プロジェクトは、重要事項説明前に必要な書類を顧客に事前送付する機能を実装することで、法令違反のリスクを解消し、顧客体験と業務効率を向上させることを目的としている。

## 2. 背景（ファクト）

### 2.1 法的要件

#### 35条書面の事前交付の条文内容

宅地建物取引士により記名された重要事項説明書及び添付書類を、重要事項の説明を受けようとする者にあらかじめ交付（電磁的方法による提供を含む。）していること。（必須）

#### 賃貸管理業法13条書面も同様の扱いとなる

**賃貸住宅管理業法 第13条第1項（重要事項説明）**

賃貸住宅管理業者は、管理受託契約を締結しようとするときは、国土交通省令で定めるところにより、あらかじめ、...（中略）...、書面を交付して説明しなければならない。

**同条 第4項（電磁的方法による提供）**

賃貸住宅管理業者は、第一項の規定による書面の交付に代えて、政令で定めるところにより、当該委託者の承諾を得て、当該書面に記載すべき事項を電磁的方法により提供することができる。この場合において、当該賃貸住宅管理業者は、当該書面を交付したものとみなす。

### 2.2 法務評価

マイページの現仕様では両重説の法令違反状態となっており、リスクスコアが20と高い状態である。なお、本事例に関して当社に顧客からクレームが発生し、契約解除となった事案も発生しており、リスクの発露可能性は顧客のリテラシーの高まり等に応じて十分発生し得る点に留意が必要。（リーガルリスクレポートより抜粋）

### 2.3 現状の業務フロー上の問題

以下のように、手付金の受領を確認→個人認証後に、初めてマイページ上で重説がダウンロードできるようになるため、条例違反状態となっている。

1. 契約準備の完了
2. 重要事項の説明　←　先に説明
3. 手付金の受領
4. このタイミングで、マイページの個人認証を実施。
5. 実施したのちに、初めて重説をマイページ上でダウンロードできる状態になっている。（ドキュサインのプレビュー画面が見れるようになる。）　←　後から交付
6. 契約

なお特定賃貸借物件のサブリース賃貸借契約書については、現状、システム的に送付するフローが存在しない。

## 3. 現状の業務フロー

### 3.1 現状の業務フロー

1. 契約準備の完了
2. 重要事項の説明（先に説明）
3. 手付金の受領
4. マイページの個人認証を実施
5. 重説をマイページ上でダウンロード可能になる（後から交付）← **法令違反**
6. 契約

### 3.2 現状の問題点

- 重要事項説明時に初めて書類を閲覧できる
- 顧客の理解度が低い
- 説明時間が長くなる（画面共有しながら説明する必要がある）
- サブリース契約書の送付フローが存在しない

## 4. ターゲットユーザー

- **主要ユーザー**: 不動産投資契約を締結する顧客
- **内部ユーザー**: 契約担当者、営業担当者

## 5. 目指す状態（Goal）

### 5.1 理想の状態

「重要事項説明書（売買、賃貸借契約）、サブリース契約書を重要事項説明よりも前に顧客に交付してあること」の要件を満たせている状態になる。

### 5.2 理想のフロー

1. 契約準備の完了
2. **契約書類準備完了をトリガーに、重要事項説明書・サブリース契約書を顧客に送付** ← **事前交付**
3. 重要事項の説明（顧客は事前に書類を確認済み）
4. 手付金の受領
5. 契約

## 6. ギャップ分析

### 6.1 現状と理想のギャップ

| 項目 | 現状 | 理想 | ギャップ |
|------|------|------|----------|
| **法令遵守** | 手付金受領→個人認証後に重説がダウンロード可能 | 重要事項説明前に重説を送付 | 法令違反のリスク |
| **書類の事前確認** | 重要事項説明時に初めて書類を閲覧 | 事前に書類を確認できる | 顧客の理解度が低い |
| **説明時間** | 画面共有しながら説明する必要がある | 事前確認済みで説明時間が短縮 | 説明時間が長い |
| **運用負荷** | 手動での事前送付は運用上不可能（月間1000契約以上） | 自動化された送付フロー | 手動運用では対応不可 |
| **サブリース契約書** | システム的に送付するフローが存在しない | 特定賃貸借物件の場合、送付フローが存在 | コンプライアンスリスク |
| **書類改変時の対応** | 再送付フローが不明確 | 電子証明書変更時に都度送付 | 対応が非効率 |
| **エラー時の対応** | エラー時の対応フローが不明確 | rollbar連携、担当営業への連携 | 対応が遅れる |

### 6.2 ギャップを埋めるための要件

- 契約書類準備完了をトリガーに、重要事項説明よりも前に書類を送付できる
- 月間1000契約以上の規模に対応できる自動化
- サブリース契約書の送付（特定賃貸借物件の場合）
- 再送付機能（電子証明書変更時）
- エラーハンドリング機能（rollbar連携、担当営業への連携）
- 受信確認機能（メールバウンス時の対応）

## 7. 課題の優先度

### 最重要

**課題1: 法令違反のリスク（最重要）**

- **現状の問題**: 手付金受領→個人認証後に重説がダウンロード可能になるため条例違反
- **影響**: リスクスコア20、免許停止のリスク、顧客からのクレーム・契約解除の事案発生
- **根拠**: リーガルリスクレポート、リスクマトリクス

### 高優先度

**課題2: 運用負荷の問題**

- **現状の問題**: 
  - 処理件数の増加: FY25で月間583件 → FY28で月間1,050件（約1.8倍）に増加予定
  - サブリース比率が下がっても処理件数は変わらないため、業務負荷は継続的に増加する見込み
  - 手動での事前送付は運用上不可能
- **前提**: **オペレーション変更で耐えようとすると負荷が高い**
  - 業務プロセス確認資料（2025/10/09）では「業務が回らなくなる」「やるがQCDに問題あり」という懸念が指摘されている
  - 「1件あたりどれくらい工数かかるか？は明らかにしておきたい。必要に応じてリソース確保や業務効率化が必要。」とされており、手動運用では対応できない規模であることが明確
  - 「カツカツの状態で今回を凌ぐのは危険だと思う」（業務プロセス確認資料より）
  - 運用プロセスを変更して対応しようとすると、以下の理由により運用負荷が高くなりすぎる：
    - 契約ごとに手動でメール送信が必要（月間1,050件以上）
    - 送付漏れのリスクが高い（人的ミスの可能性）
    - エラー時の対応が非効率（個別対応が必要）
    - 書類改変時の再送付が手動対応になる（都度対応が必要）
    - 監査証跡の管理が困難（手動記録が必要）
- **影響**: 手動運用では対応できない規模、**システム的な自動化が必須**
- **根拠**: 
  - プロジェクト概要書のFAQで明確に言及されている
  - 業務プロセス確認資料（【業務プロセス確認】サブリース重説に特約.pdf）で具体的な懸念が指摘されている

**課題3: サブリース契約書の送付フローが存在しない**

- **現状の問題**: 特定賃貸借物件のサブリース賃貸借契約書については、現状、システム的に送付するフローが存在しない
- **影響**: 特定賃貸借物件の顧客開示ができない、コンプライアンスリスク
- **根拠**: PJ書の背景（ファクト）で明確に言及、リスクマトリクスで高リスクと評価

**課題4: 書類の事前確認ができない**

- **現状の問題**: 重要事項説明時に初めて書類を閲覧
- **影響**: 顧客の理解度が低い
- **根拠**: 現状の業務フロー

**課題5: 説明時間の非効率性**

- **現状の問題**: 画面共有しながら説明する必要がある
- **影響**: 説明時間が長くなる
- **根拠**: 現状の業務フロー

### 中優先度

**課題6: 書類改変時の再送付フローの課題**

- **現状の問題**: 契約書・重説がReadyとなった後、改変が生じた場合の再送付フローが不明確
- **影響**: 書類改変時の対応が非効率、顧客への再送付が手動対応になる可能性
- **根拠**: FAQで言及されている

**課題7: エラー時の対応フローの課題**

- **現状の問題**: メール送信エラー時の対応フローが不明確、メールバウンス時の担当営業への連携方法が未整備
- **影響**: エラー発生時の対応が遅れる、顧客への書類送付が漏れるリスク
- **根拠**: 開発内容でエラー時の対応方針が記載されているが、現状の課題として明確化されていない

**課題8: 顧客がメールを見落とすリスク**

- **現状の問題**: 書類をメールで送付した場合、顧客がメールを見落とす可能性がある
- **影響**: 書類が届いていないと顧客が認識する可能性、重要事項説明の準備ができない
- **根拠**: ソリューション定義書のリスクとして言及されているが、現状の課題として明確化されていない

### 低優先度

**課題9: 送付タイミングの課題**

- **現状の問題**: 契約書類準備完了のタイミングが適切かどうかの検証が必要
- **影響**: 送付タイミングが早すぎる・遅すぎる可能性、顧客が十分な検討時間を確保できない可能性
- **根拠**: 開発内容で「契約書類準備完了をトリガーにする」と記載されているが、現状の課題として明確化されていない
